# 设计 spec:协作执行命令 `do` + 三功能厘清(cc-codex-review)

- 日期:2026-06-19
- 状态:**已批准(命名 `do`、繁简自适应已确认),待写实现计划**
- 关联:DESIGN §12;extract-reqs spec(2026-06-12);"A 反锚定/独立首判"

## 1. 背景
当前只有 `review`(事后审)+ `extract-reqs`(提需求),缺最自然的入口:"给个任务,让两个 AI 协作做掉"。补这个入口 `do`,并把三命令的职责厘清成一条流水线,避免再"融合"。

## 2. 三命令职责
| 命令 | 时机 | 干什么 | 产出 |
|---|---|---|---|
| **`do`** | 做之中 | 你给任务(问答 / 动手),Claude+Codex 协作做掉 | 做完的东西(+ 待你拍板点) |
| **`review`** | 做之后 | 审已完成的产物 / diff / 方案 | RESOLVED / UNRESOLVED 裁决 |
| **`extract-reqs`** | 中间锚 | 把你认可的需求固化成基准 | 用户认证需求文件 |

## 3. `do` 详细设计

### 定位
协作版的"我说、Claude 做":你给任意任务(问答,或动手如"建 111.txt 写 111""做个人主页 信息 xxx""实现 xxx,有问题与我讨论"),Claude 和 Codex 协作完成 —— **不只是出方案,该做就做**。

### 角色铁律
- **Claude = 唯一动手方**:写文件、实现,都由 Claude 做。
- **Codex = 只读协作方**:出方案、挑刺、复核 Claude 做的;**绝不能写文件**(工具硬不变量,沿用现有只读沙箱)。

### 流程
1. **歧义先问**:任务/需求有重大歧义 → 先反问用户澄清,再继续。
2. **繁简自适应**(关键):Claude 据任务判断复杂度——
   - **琐碎且明确**(如建个文件):Claude 直接做,Codex 快速复核(或免),不折腾;
   - **复杂 / 有歧义 / 有方案空间**(如做主页、实现功能):走完整协作(下面 3–5)。
3. **独立出方案**:Claude 和 Codex **各自独立**基于需求出方案(**互不看对方方案**,消除锚定)。
4. **对抗到统一**:两方案对抗讨论收敛到统一(默认 `--max-rounds 3`);谈不拢 → **综合"双方已一致的方案骨架" + 列"仍分歧的几处(各自主张)"** 给用户,不假装统一。
5. **执行 + 复核**:Claude 按统一方案动手做;Codex 复核做出来的结果(只读)。
6. **产出**:做完的东西 + 待你拍板的点。

## 4. `review`(已有,微调)
行为不变;**默认 `--max-rounds` 由 5 改为 3**(质量优先,非为收敛而收敛;想深挖再加 flag)。

## 5. `extract-reqs`(已有,微调)
行为不变;**新增"对话深度"可选参数**(看最近 N 次对话);不指定则 Claude 自己判断从哪开始算需求。

## 6. 贯穿三命令的统一规则
- **轮数**:都用 `--max-rounds`,**默认 3**(`0`=无上限);每次开始**回显一行**"本次最多 N 轮(来源:默认/flag)";到顶未收敛**提示可加轮**。不弹窗打断。
- **不收敛 ≠ 失败**:到顶/僵持都产出结构化结果 —— **综合"已达成一致的部分" + 列"仍分歧的"**(各自立场 + 严重度),质量优先。
- **防无限轮**(沿用现有):硬上限 + 停滞检测;只有显式 `--max-rounds 0` 才无上限。
