# 设计 spec:需求提取命令 `extract-reqs`(cc-codex-review 扩展)

- 日期:2026-06-12
- 状态:**已 brainstorm 批准,待写实现计划**
- 关联:DESIGN §12;与"A 反锚定/独立首判"同源互补(见 §1)

## 1. 背景与问题
cc-codex-review 让 Claude(驱动/实现方)与 Codex(独立 reviewer)互审。在「当前窗口开发」场景,三方角色:
- **用户** = 开发人员,给需求、定方向,**需求的权威持有者**(真需求以用户为准);
- **Claude** = ① 实现者(写代码,对自己实现有"作者偏向") ② review 驱动者;
- **Codex** = 独立 reviewer(没参与实现、没看过用户与 Claude 的对话)。

评审须对照"需求"判断工作是否合格。但现状(review.md §2/§4)里,Codex 拿到的「任务目标」是 **Claude 从对话里单方转述**的——Codex 看不到原始对话。由此两个泄漏:
- **转述损耗**:Claude 可能漏传 / 传歪需求;
- **私货夹带**:Claude 的设计决策(HOW)可能被当成需求(WHAT)混入基准,使评审退化为"在 Claude 定义的需求内自证",**发现不了方向跑偏**(最严重的一类问题)。

这与"结论锚定"(A:Claude 把结论塞给 Codex 带偏它)**同源**——根都是「评审的输入经作者一手过滤」。
> **extract-reqs 负责把"需求基准"这一刀剥干净;A(反锚定)负责剥"结论"那一刀。**

## 2. 目标
提供一个命令:从当前会话对话中提取**经用户背书的需求(纯 WHAT)**,经用户确认后产出「用户认证需求」,供 review 用作评审基准——**不夹带 Claude 未经确认的设计**。

## 3. 命令接口
`/cc-codex-review:extract-reqs [界定指令]`(界定指令**可选**;也支持中文自然语言触发,如「帮我提取需求」——触发词是底层 skill 的事,用户无需记)
- **不带参数(默认)**:Claude **主动**从当前会话归纳「用户最近让我做的需求」并提取;**提取范围由 Claude 判断,遇多块/范围模糊时先问用户确认**(不瞎猜、不全量乱提)。这是主用法——用户不必先想清楚范围。
- **带 `[界定指令]`(可选)**:用户给范围指示(如「游戏 UI 相关需求」),Claude 按它界定。
- 可选 `--out <path>`:产物输出路径(不给则默认写临时文件 + 在对话内呈现)。

## 4. 过程
1. **定范围**(支持无参数主动提取):
   - 给了 `[界定指令]` → 以它为范围;
   - **没给** → Claude 扫描当前会话、归纳"最近的需求活动":**单一清晰块** → 直接取;**多块 / 范围模糊** → **先问用户确认范围**(给候选,如「是指刚才的游戏 UI 那块,还是整个会话的需求?」),再继续。
   - 原则:**宁可问,不瞎猜、不全量乱提**(提多了噪音,提错了漏需求)。会话很长可能有上下文压缩,归纳基于**可见部分**,并提示用户"更早的需求若没覆盖请补"。
2. **扫描提取**:在确定的范围内,识别需求点。
3. **分档**(判据 = **是否经用户背书**,**不是**谁先提出):

   | 档 | 内容 | 出处要求 |
   |---|---|---|
   | 纳入① | 用户**直接提**的需求/约束 | 附用户原话引用 |
   | 纳入② | 源自 Claude、用户**明确同意**的论断 | **必附"用户表示同意"的那句原话** |
   | 待定③ | Claude 提的、用户**未明确表态**(仅默许/未反对) | 附 Claude 提议原文 |
   | 排除④ | Claude **单方**实现/设计细节(用户未参与) | 列出供核对"排除是否正确" |

4. **硬规则(fail-closed,防私货洗白)**:②**必须**附得出"用户明确同意"的原话;**附不出则自动降级③**。即无用户明确背书,不得擅自计入需求。
5. **用户确认**:用户对清单 改 / 增 / 删,并勾选③中哪些计入需求。
6. **产出**:一份「用户认证需求」markdown(仅 WHAT:纳入①② + 用户勾选的③),每条带出处;排除④单列为"非需求(被审对象)"附录。

## 5. 产物格式(用户认证需求文件)
```
# 用户认证需求:<范围描述(用户的界定指令,或 Claude 自动归纳并经用户确认的范围)>
（生成时间 · 来源会话标记 · "经用户确认 ✓" 标记）

## 需求(评审基准 · 纯 WHAT)
- [R1] <需求点>  · 出处:<用户原话>        · 来源:用户直述
- [R2] <需求点>  · 出处:<用户同意的原话>  · 来源:Claude 提议 + 用户同意
- ...

## 附录:已排除项(非需求 / 属被审的实现决策)
- <Claude 单方设计>  · 原因:用户未背书
```

## 6. 与 review 的关系
- **MVP(本 spec)**:**解耦**。产物经 `review --plan <文件>` 引用,作为「任务目标」基准。
- **演进(未来,非本 spec)**:review 默认先调 extract-reqs + 用户确认再审(方案 3);已给 `--plan` 或 `--skip-extract` 则跳过。**需 MVP 验证提取质量后再做。**

## 7. 形态与实现要点
- **prompt 级命令**(`commands/extract-reqs.md`):Claude 读对话 → 分档 → 呈现 → 产出。与 review 一样由 prompt 驱动,**不需要新的确定性脚本**(分档是语义判断 + 用户确认兜底,无可单测的确定性纯逻辑)。
- 写产物文件由命令负责;**只写产物文件,不触碰其他文件**。

## 8. 验收标准
给定一段含「用户需求 + Claude 设计 + 用户对部分设计的同意/默许」的对话,extract-reqs 能:
- **无参数调用**时自动归纳提取范围;范围多块/模糊时**主动问用户**(而非瞎猜或全量乱提);给了界定指令时按指令界定;
- 把用户直述需求归①、用户明确同意的 Claude 论断归②(附同意原话)、未表态的归③、Claude 单方设计归④;
- ②无同意原话时降级③(硬规则生效);
- 产物仅含 WHAT、每条带出处、排除项单列;
- 用户可改/勾选后产出确认版;
- `review --plan` 能消费该产物作为评审基准。

## 9. 局限(诚实标注)
- 只基于**当前会话可见上下文**;需求若在已压缩/更早会话里,可能漏。
- 分档是 Claude 的语义判断,**靠"用户确认"兜底**——用户不认真核对仍可能放过误分类(与工具一贯"人工兜底"一致)。
- prompt 级、无单测;靠手动 / 用户确认验收(同 lens 的判断型部分)。

## 10. YAGNI(不做)
- 不做 review 内置一条龙(演进项);
- 不跨会话翻历史;
- 不做需求的版本管理 / diff(产物就是一次性 markdown)。
